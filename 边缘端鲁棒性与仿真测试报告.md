# 中央空调 AI 寻优系统 —— 边缘端鲁棒性与仿真测试报告

> 范围：Cursor 负责的核心算法与鲁棒逻辑（能耗模型、约束校验、数据清洗、
> PSO 寻优、熔断兜底、高仿真模拟）。云端不在本次测试范围内。

## 1. 测试目的

在纯模拟数据上对边缘端核心链路做**全方位对抗性测试**，投喂一切可以想到的
错误数据与异常工况（不限于需求文档列举项），验证系统在任意输入下均满足以下
**核心安全不变量**：

1. 绝不抛出未捕获异常、绝不崩溃；
2. 下发的控制参数**始终有限且满足设备安全硬约束**；
3. 能耗 / 节能率等输出数值**始终有限、语义合法**（非负、非 NaN/Inf）；
4. 熔断、兜底、阶梯平滑、工况突变自适应等鲁棒机制按预期动作；
5. 关键传感器持续失效时，系统**安全降级到固定参数，绝不失控**。

## 2. 测试方法与环境

- **方法**：单元测试 + 参数化对抗测试 + 大样本随机 fuzz 仿真 + 场景专项压测。
- **数据来源**：`HospitalDataGenerator` 高仿真度医院空调时序模拟（物理自洽，
  含极端场景注入）。
- **环境**：Python 3.14，pytest 8.x，scikit-opt 0.6.6 + numpy + scipy。
- **测试资产**：
  - `edge/tests/test_algorithms.py`（31 项，核心功能与机制）
  - `edge/tests/test_robustness.py`（52 项，对抗性与异常场景）
  - `edge/tests/test_api_data_status_logging.py`（5 项，数据/状态 API 与日志落库）
  - `edge/tools/simulation_report.py`（大样本仿真报告生成器）

**结果总览：全部 97 项测试通过（含 Trae 框架既有 9 项）。**

## 4. 测试中发现的问题与修复

本轮对抗测试**实测暴露出 10 个真实缺陷**（全部集中在非有限值处理），均已修复。

### 4.1 严重：`NaN/Inf` 被约束校验误判为“合法”

- **现象**：`SafetyConstraints.validate` 用 `value < lo or value > hi` 判越界，
  而 `NaN` 与任何数比较均为 `False`，导致 **NaN 参数被判定为合法**并可能下发。
- **风险**：向 DDC 下发 NaN 设定值 → 现场设备行为不可预测（医疗场景致命）。
- **修复**：`validate` / `clip` / `hard_violation` / `comfort_penalty` 全部显式
  拦截非有限值——校验直接判非法、裁剪回退区间中值、惩罚记为极大有限值。

### 4.2 严重：`NaN/Inf` 输入沿能耗模型传播为 NaN 总能耗

- **现象**：负荷/温度/湿度/控制参数中任一为 `NaN/Inf` 时，`ACEnergyModel`
  输出 `total_power=NaN`，进而污染 PSO 目标值、节能率、下发参数与落库记录。
- **修复**：能耗模型入口统一净化 `_finite(value, default)`，非有限值回退安全
  默认；负荷额外做非负与上限（1e7）保护，保证任意输入下输出恒为有限。

### 4.3 相关联问题：寻优结果 `predicted_power` 为 NaN

- 由 4.1 / 4.2 连带产生，随上述修复一并消除。修复后寻优输出在所有对抗输入下
  `predicted_power`、`energy_saving_rate` 均有限且非负。

> 说明：上一轮 code review 已修复的两个**核心逻辑**问题（① 持续工况突变被永久
> 误杀 → 新增工况突变自适应；② 阶梯平滑跟不上负荷突变 → 新增受限应急平滑）
> 在本轮仿真中得到正面验证（见 §5 的 `regime_shifts` 与应急平滑触发）。

### 4.4 接口与存储一致性问题（本轮补充修复）

根据 API/数据层复查，本轮新增修复 5 个工程集成问题：

1. `/api/v1/data/realtime` 返回结构不一致：现在无论走数据库还是模拟器回退，
   都统一返回 `raw_data` 为 `dict`，并同时返回常用扁平字段，客户端不再需要二次
   `JSON.parse()`。
2. 新增 `/api/v1/data/history?page=1&page_size=50`，支持运行工况分页查询；
   单页上限 500，避免一次性拉取过大。
3. 新增 `/api/v1/status/local`，并让静态状态页调用该路径，避免状态页/巡检工具
   访问路径不统一导致 404。
4. `runtime_data` 新增常用结构化字段（室外温度、室内负荷、总功率、冷水温度等），
   `save_runtime_data()` 保存时自动从 `raw_data` 填充；`init_db()` 内置幂等 SQLite
   迁移并回填既有历史数据。
5. `alarm_log` / `operation_log` 已接入业务：手动寻优、定时寻优、模拟数据生成、
   数据异常、寻优失败、约束失败、任务异常等路径都会写操作日志或告警。

## 6. 结论

- **核心安全不变量在全部 2800+ 仿真周期与 97 项测试中零违反**：无崩溃、无越界
  下发、无 NaN/Inf 输出。
- 需求文档要求的五大鲁棒性（数据 / 算法 / 设备 / 架构 / 业务）均有对应测试证据。
- 测试实测发现并修复了 10 个非有限值处理缺陷，并补齐 API/历史查询/结构化存储/
  告警审计等 5 个工程集成问题，显著提升了工业级健壮性与可运维性。
- 寻优在满足设备安全与医疗舒适约束的前提下稳定给出节能方向，性能满足边缘端
  实时要求（单次寻优约 15ms，远低于 10~15 分钟寻优周期）。

## 7. 遗留说明与后续建议（非缺陷）

1. **能耗模型现场标定**：额定功率、卡诺效率系数、供冷能力等需按现场装机容量
   校准；建议对接真实历史数据回归修正后再定节能承诺值。
2. **多机组逐台并联**：当前为“等效机组”聚合建模，若需精细到每台机组/水泵的
   启停组合寻优，可扩展为多机组模型。
3. **告警落库接入**：熔断/兜底/工况突变事件已有日志，建议在调度层写入
   `alarm_log`（编排层，属框架侧），便于云端审计与大屏展示。
